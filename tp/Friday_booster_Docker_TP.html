Quelques rappels

Docker est un outil qui peut empaqueter une application et ses dépendances dans un conteneur isolé, qui pourra être exécuté sur n’importe quel serveur.

Une image Docker est une représentation statique de l’application ou du service, de leur configuration et de leurs dépendances.

Un conteneur correspond à l’instanciation d’une image docker. C’est-à-dire, selon l’image instanciée, qu’un conteneur correspond à une instance d’un système d’exploitation contenant des produits et des livrables (binaires, configuration).

Principales commandes

Ce premier exercice a pour objectif de vous aider à appréhender les principales commandes Docker :

Table 1. Principales commandes
Commande Description

docker pull <nom_image[:tag]>

Télécharge l’image indiquée (sur le Docker Hub), avec possibilité de préciser le tag (la version).

docker run [-d] [-p X:Y] <nom_image>

Instancie un conteneur à partir de l’image indiquée.

Remarque : si l’image n’est pas disponible localement, un « docker pull » est automatiquement exécuté.

  • L’option « -d » permet de lancer le conteneur en tâche de fond et rendre la main à l’utilisateur.

  • L’option « -p X:Y » permet de déclarer une redirection du port X du serveur hôte (du nœud) vers le port Y du conteneur.

docker ps

Liste les conteneurs instanciés, leur état, etc.

docker start/stop <UID_ou_début_UID>

Démarre ou arrête un conteneur (sans le supprimer).

Remarque : il n’est pas nécessaire de saisir l’IUD en entier, les 2/3 premiers caractères de l’UID suffisent (s’il n’y a pas d’ambiguïté entre plusieurs conteneurs).

docker rm <UID_ou_début_UID>

Supprime un conteneur.

Remarque : il faut préalablement stopper un conteneur pour le supprimer.

docker attach <UID_ou_début_UID>

Dans le cas où le conteneur a été lancé en tâche de fond (option -d), ouvre la console du conteneur.

Pour détacher la console, utiliser le raccourci CTRL+C.

docker exec –it <UID_ou_début_UID> <commande>

Exécute une commande dans un conteneur. Il peut être utile d’ouvrir un invité de commande « dans le conteneur » via la commande docker exec –it <UID_ou_début_UID> /bin/bash . Taper « exit » pour sortir du conteneur.

Créez votre propre image

Dans cet exercice, l’objectif est d’instancier vos propres images Docker à partir d’images standards issues du DockerHub et des ressources du TP.

Il s’agit de créer une image tomcat-sample à partir de l’image tomcat:jdk8 en y ajoutant un fichier WAR (application Web JAVA) puis d’instancier un conteneur à partir de cette image.

Pour cela:

  • Inspecter le contenu du répertoire dockerfile.

  • Compléter le fichier Dockerfile, suivant les mentions TODO qu’il contient.

Dockerfile
#TODO : Se baser sur l'image tomcat:jdk8

#TODO : Déposer le fichier sample.war dans le répertoire webapps de tomcat : /usr/local/tomcat/webapps/
Aide 
Utiliser le mots-clé « FROM » pour indiquer l’image d’origine.
Aide 
Utiliser le mots-clé « COPY » pour déposer le fichier WAR sous /usr/local/tomcat/webapps/.
Solution 
#TODO : Se baser sur l'image tomcat:jdk8
FROM tomcat:jdk8
#TODO : Déposer le fichier sample.war dans le répertoire webapps de tomcat : /usr/local/tomcat/webapps/
COPY ./sample.war /usr/local/tomcat/webapps/

Par la suite:

  • A l’aide de la commande docker build, créer à partir du Dockerfile une nouvelle image dont le nom est tomcat-sample.

Et maintenant, une stack avec compose !

Docker Compose est un outil utilisé pour définir et exécuter des applications Docker multi-conteneurs. il remplace les commandes docker run qui sont exécutées à plusieurs reprises sur chaque conteneur dans une application multi-conteneurs.

En utilisant Docker Compose, nous pouvons simplifier l’exécution d’applications multi-conteneurs à seulement deux commandes : docker-compose up et docker-compose down.

Dans cet exercice, vous apprendrez comment Docker Compose facilite la configuration et l’exécution de plusieurs conteneurs Docker. À des fins de démonstration, on utilisera des conteneurs Spring Boot et PostgreSQL.

Dockeriser le projet Spring Boot friday-booster

  • Builder l’image docker.

CMD
$ docker build -t friday-booster:latest .
Vous pouvez exécuter l'image Docker du projet Spring Boot friday-booster à l'aide de la commande $docker run, Mais rappelez-vous, nous voulons utiliser Docker Compose pour démarrer tous les services du projet.

Définir le fichier Docker Compose

Maintenant que vous avez dockerisé le projet Spring Boot friday-booster , vous devez définir un fichier docker-compose.yml
Ce fichier aura une configuration pour créer une base de données PostgreSQL dans un conteneur Docker et la connecter au conteneur du projet Spring Boot friday-booster pour s’exécuter en tant qu’application multi-conteneurs.

En règle générale, lors de la configuration d'un projet Spring Boot qui se connecte à une source de données, nous utilisons des configurations similaires à celle illustrée ci-dessous dans le fichier application.properties/application.yml du projet :
spring.datasource.url = jdbc:postgresql://localhost:5432/fridaybooster
spring.datasource.username = postgres
spring.datasource.password = postgres
Les configurations ci-dessus donneront à Spring Boot un accès à une base de données PostgreSQL. Dans cet exercice, vous allez configurer les configurations ci-dessus avec un fichier
docker-compose.yml

  • Se baser sur la structure suivante pour rédiger le fichier docker-compose.yml

docker-compose.yml
version: "3.8"
services:
#======================================
#======= friday-booster service =======
#======================================
  friday-booster:

#==================================
#======= PostgreSQL service =======
#==================================
  PostgreSQL:

#========================
#======= networks =======
#========================
networks:
  friday_booster_network:
#=======================
#======= volumes =======
#=======================
volumes:
  db-data:
  log-data:
Les configurations ci-dessus définissent la version de Docker Compose et les services dans l'application multi-conteneurs ainsi qu'un networks ayant comme nom "friday_booster_network".
